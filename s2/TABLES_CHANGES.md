# Описание изменений схемы и генерации данных

### 1. Таблица `client`
*   **Объем:** 260 000 строк.
*   `phone_number` — для высокой кардинальности/селективности (все телефоны уникальны).
*   `home_address_coords` (тип `POINT`) — геометрический тип, NULL в 15% случаев
*   `notes` (тип `TEXT`) — для полнотекстового поиска, заполняется случайным текстом.
*   `registration_date` — равномерное распределение (за последние 5 лет).

### 2. Таблица `passenger`
*   **Объем:** 260 000 строк.
*   `profile_data` (тип `JSONB`) — хранит предпочтения, NULL в 20% случаев
*   `gender` — для низкой селективности (всего 2 значения 'M'/'F', распределение 50/50).

### 3. Таблица `flight`
*   **Объем:** 260 000 строк.
*   `flight_tags` (тип `TEXT[]`, массив) — теги рейса (`wifi`, `meal`). Длина массива от 0 до 3.
*   `booking_window` (тип `TSTZRANGE`, range) — диапазон времени, когда открыта продажа билетов.

### 4. Таблица `booking`
*   **Объем:** 500 000 строк.
*   `channel` — низкая кардинальность (всего 4 значения: WEB, APP, PARTNER, OFFLINE).
*   `client_id` — здесь распределение Ципфа (Zipf):
    - Суть: есть топ клиентов, которые делают тысячи заказов.
    - 70% записей принадлежат 10% значений
