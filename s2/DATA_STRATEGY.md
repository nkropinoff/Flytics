# Data Generation Strategy & Requirements Matrix

## 1. Целевые таблицы и объемы
**Требование:** Общий объем > 1 млн строк, не менее 250,000 строк в каждой таблице.

| Таблица | Целевой объем | Роль в тесте |
| :--- | :--- | :--- |
| **`client`** | 250,000 строк | Master-данные, геометрия, полнотекст |
| **`passenger`** | 250,000 строк | Детализация, JSONB, коррелированные данные |
| **`flight`** | 250,000 строк | Временные ряды (Time Series), диапазоны, массивы |
| **`booking`** | 500,000 строк | Транзакционные данные, Zipf-распределение, связи |

---

## 2. Реализация требований по типам данных (Types Requirements)
**Требование:** Наличие специфических типов (JSONB, Arrays, Geometry, Range, Fulltext).

| Требование | Поле (Таблица) | Тип данных | Описание генерации |
| :--- | :--- | :--- | :--- |
| **JSONB** | `passenger.profile_data` | `JSONB` | Структура: `{"meal": "vegan", "seat": "aisle", "priority": true}`. Значения выбираются случайно из словаря (`standard/kosher/halal`, `window/aisle`). |
| **Массивы** | `flight.flight_tags` | `TEXT[]` | Набор тегов: `['wifi', 'meal', 'business_class']`. Длина массива от 0 до 3 элементов. |
| **Геометрия** | `client.home_address_coords` | `POINT` | Координаты (lat, lon). Генерируются кластерами вокруг крупных городов (Москва, Казань) с небольшим разбросом. |
| **Range-типы** | `flight.booking_window` | `TSTZRANGE` | Диапазон времени продаж: `[departure - 3 months, departure - 1 hour)`. |
| **Полнотекст** | `client.notes` | `TEXT` | Генерация предложений разной длины (Lorem Ipsum) для проверки индексации `tsvector`. |

---

## 3. Реализация требований по распределению (Distribution Requirements)
**Требование:** Использовать разные типы распределений и кардинальности.

### 3.1. Кардинальность (Uniqueness / Selectivity)
| Тип | Поле | Стратегия | Цель / Почему это важно |
| :--- | :--- | :--- | :--- |
| **Высокая (High)**<br>*(~100% уникальных)* | `client.phone_number` | Генерация уникальных телефонов `+79XXXXXXXXX`. 100% уникальность. | Проверка эффективности **B-Tree** индексов (Index Scan). |
| **Низкая (Low)**<br>*(3-5 значений)* | `booking.channel` | **4 значения:** `'WEB'`, `'APP'`, `'PARTNER'`, `'OFFLINE'`. Распределение ~25% на каждое. | Проверка поведения Planner (игнорирование индекса при >10% строк). |
| **Низкая (Low)**<br>*(3-5 значений)* | `passenger.gender` | 2 значения: `'M'`, `'F'`. Распределение 50/50. | Анализ селективности предиката. |

### 3.2. Статистические распределения
| Тип | Поле | Стратегия (Python) | Цель / Реализация "Перекоса" |
| :--- | :--- | :--- | :--- |
| **Zipf (Skewed)**<br>*(Сильно неравномерное)* | `booking.client_id` | **Закон Ципфа (Zipfian law).** Параметр `exponent=1.5`.<br>Результат: **Топ-10 клиентов** делают тысячи заказов (турагенты), "хвост" делает по 1 заказу. | Выполнение требования *"70% записей принадлежат 10% значений"*. Тест кэширования (Hot pages). |
| **Uniform**<br>*(Равномерное)* | `client.registration_date` | Случайная дата в диапазоне `[NOW() - 5 years, NOW()]`. Вероятность попадания в любой день одинакова. | Базовый сценарий для гистограмм статистики. |
| **Normal**<br>*(Гауссовское)* | - | (Косвенно используется при генерации координат геометрии вокруг центров городов). | Реалистичное распределение гео-данных. |

## 4. Требования по NULL и "частичным данным"
**Требование:** NULL-значения в 5–20% строк.

| Поле | % NULL (Пустых) | % Заполненных | Обоснование |
| :--- | :--- | :--- | :--- |
| `passenger.profile_data` | **20%** | 80% | У большинства пассажиров есть предпочтения, но у части — нет. |
| `client.home_address_coords` | **15%** | 85% | Большинство указали адрес, но 15% скрыли. Идеально для **Partial Index** (`WHERE coords IS NOT NULL`). |
| `client.notes` | **10%** | 90% | Заметки есть почти у всех, но иногда отсутствуют. |
| `flight.flight_tags` | **5%** | 95% | У 5% рейсов нет никаких особенностей (пустой массив или NULL). |

---

## 5. Порядок наливки (Dependency Order)
Строгий порядок генерации для соблюдения целостности (Foreign Keys):

1.  **`client`** (250k) — Генерируем базу пользователей. Сохраняем их ID в памяти.
2.  **`passenger`** (250k) — Генерируем пассажиров (независимо, для массовости).
3.  **`flight`** (250k) — Генерируем рейсы, ссылаясь на существующие `aircraft` и `flight_number` (из справочников V1).
4.  **`booking`** (500k) — **Связующее звено.**
    *   Берем ID существующих клиентов (используя Zipf распределение).
    *   Создаем заказ.
5.  *(Автоматически)* **`ticket`** / **`payment`** — (Опционально) Создаются как следствие бронирования (для полноты картины).
